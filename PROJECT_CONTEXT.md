# bbcare 项目说明（面向自己与 AI）

> 这份文档不是 README，也不是对外宣传材料。
> 它更像是 `项目导读 + 架构速览 + 维护建议 + AI 接手上下文`，目的是让自己或其他 AI 在较短时间内理解 bbcare 现在“已经是什么、为什么这么做、哪里还值得改”。

## 1. 项目一句话概述

`bbcare` 是一个 **移动端优先的孕期陪伴小应用**，核心是 **胎动记录**，并逐步扩展出 **体重管理、孕周进度、每日菜单、孕妇饮食拍照问 AI、金价查询、数据备份恢复、基础日志与容器部署** 等能力。

它不是一个“严格医疗系统”，而是一个 **偏个人/家庭使用的生活管理工具**。当前代码形态更接近：

- 一个单页前端应用；
- 一个轻量 Node/Express 后端；
- 一个单文件 SQLite 存储；
- 若干面向个人使用场景的小工具；
- 部分 AI 能力通过智谱 / 阿里云百炼接入。

## 2. 项目定位与目标用户

### 2.1 主要定位

项目当前最适合的定位是：

- **个人自用或家庭内部使用的孕期记录工具**；
- **以手机浏览器访问为主的轻应用**；
- **偏“记录 + 参考 + 辅助判断”**，而不是医疗诊断平台。

### 2.2 主要用户

更像在服务下面几类人：

- 孕妇本人：记录胎动、查看趋势、管理体重；
- 家人/伴侣：帮助记录与查看；
- 开发者本人：把它当作一个长期演进的小型生活产品；
- AI 助手：理解项目结构、继续维护、补功能、排查问题。

### 2.3 当前产品气质

从 UI 和实现方式看，这个项目有几个明显特征：

- **强移动端导向**：底部导航、卡片化布局、手势下拉刷新、较明显的手机视觉风格；
- **单人产品感很强**：很多设计都以“一个家庭/一个部署实例”的使用方式为默认前提；
- **产品边界偏灵活**：胎动是核心，但工具页已经开始承载饮食、菜单、金价等扩展功能；
- **工程上追求轻量**：依赖很少，没有引入大型状态管理、ORM、复杂路由或组件库。

## 3. 项目现在到底做了什么

## 3.1 首页：胎动记录主工作台

首页是项目最核心的页面，承担“打开就能用”的主任务。

当前能力包括：

- 展示孕周、孕期阶段、预产期进度；
- 展示最近体重，并支持直接输入“今天体重”；
- 通过 **三个漂浮气泡按钮** 快速记录胎动；
- 胎动可区分动作类型：
  - `hiccup`：打嗝
  - `hand`：伸手 / 轻柔动作
  - `kick`：踢脚 / 明显动作
- 统计当前小时：
  - 原始点击数；
  - 有效胎动数；
- 展示最近按小时聚合的胎动记录；
- 点击某小时可展开详情，查看该小时内的具体事件列表；
- 支持首页手势下拉刷新，与服务端状态对齐。

### 3.1.1 “有效胎动”规则

这部分是领域规则里的关键：

- 不是每次点击都算一次有效胎动；
- **5 分钟内的多次点击视为 1 次有效胎动**；
- 统计趋势时，按这个规则进行“去重”。

这说明当前项目在胎动统计上，已经不是纯粹“点击计数器”，而是有一个轻量的业务规则模型。

### 3.1.2 首页交互体验特点

首页明显是花过心思的：

- 气泡有持续漂浮动画；
- 气泡速度可配置；
- 有触摸下拉刷新；
- 风格偏 iOS / 玻璃拟态 / 深色主题友好；
- 适合手机单手快速记录。

也就是说，首页不是“表单页”，而是整个产品体验的门面。

## 3.2 统计页：趋势、日历、体重管理

统计页负责把“记录”变成“可回看、可解释”的内容。

当前包括几块：

### 3.2.1 12 小时有效胎动趋势

- 统计过去 12 小时的有效胎动；
- 按小时展示；
- 每小时有效胎动 `>= 3` 视为“正常”，否则“偏少”；
- 可展开查看每小时明细。

这里的定位不是医学结论，而是 **观察趋势**。代码里文案也比较克制，没有直接当成诊断结论。

### 3.2.2 日历视图

- 以日历形式展示历史天数据；
- 进入某一天后，可打开抽屉查看当天：
  - 总有效胎动；
  - 总点击数；
  - 是否达标；
  - 当天全部事件；
- 还支持从“日抽屉”里继续跳到某个小时详情。

这个设计很好，因为它把“日维度”和“小时维度”打通了。

### 3.2.3 体重趋势图

- 记录孕期体重；
- 展示最近体重与前一条体重对比；
- 统计页里有曲线图；
- 支持按日期查询、编辑、删除体重记录；
- 同一天重复录入会覆盖旧值。

### 3.2.4 体重记录管理

这块更偏“工具性后台”，对长期使用很重要：

- 可以新增；
- 可以按日期查询；
- 可以删除已有记录；
- 会做日期与数值规范化。

## 3.3 小工具页：项目的扩展能力集中地

小工具页已经不是单一能力，而是一个“小型功能集市”。

### 3.3.1 每日菜单

这是一个相当有意思的能力：

- 菜单来自一个 **16 周食谱数据集**；
- 数据当前已经固化为 `server/daily_menu_data.json`；
- 后端按“日期 → 周次/星期 → 菜单条目”映射返回；
- 前端支持切换日期查看；
- 每条菜单包含：
  - 餐次；
  - 食物；
  - 生重；
  - 碳水 / 蛋白 / 脂肪 / 膳食纤维；
  - 热量；
  - 进食顺序提示。

#### 当前菜单映射规则

现在的规则不是“按真实孕周强绑定”，而是：

- 取一个锚点日期；
- 以 **Week1 的周四** 对齐“今天”；
- 再根据用户选中的日期做循环偏移；
- 最终落到 16 周 × 7 天的一个循环菜单上。

这意味着：

- 当前菜单更像“一个循环食谱参考库”；
- 不是严格根据孕妇真实孕周自动推荐；
- 但它足够轻便，且很适合先把食谱工具做出来。

### 3.3.2 菜单 AI 推荐菜

每日菜单不是静态展示结束，还支持把某个餐次/食材发给 AI，让 AI 推荐菜。

能力特点：

- 输入包括：餐次、现有食材、日期、星期、Week 编号；
- 可选智谱 / 阿里云百炼两个供应商；
- 使用单独的菜单推荐 system prompt / user prompt；
- 有重试、超时与错误友好提示；
- 更偏“从给定菜单继续延展”的轻问答，而不是完整营养计划生成器。

### 3.3.3 孕妇饮食拍照问 AI

这是当前扩展能力里最“产品化”的一个点。

当前流程：

1. 用户从相册上传或拍照；
2. 前端压缩图片、生成预览；
3. 根据配置选择两种提交模式：
   - `url`：先上传到服务端，再把图片 URL 发给模型；
   - `inline`：直接把图片内容以内联方式提交；
4. AI 返回：
   - 这是什么食物/菜品；
   - 孕妇能不能吃；
   - 如果不建议，原因是什么；
   - 如果可以，营养价值与推荐量是什么。

体验上还做了这些细节：

- 图片压缩目标可调；
- 有进度与耗时展示；
- 支持再测一次；
- 支持自定义 system prompt / user prompt；
- 支持 thinking 开关；
- 会打客户端日志到后端。

### 3.3.4 今日金价

当前金价工具更像“顺手做得不错的生活工具”：

- 展示实物黄金、金条、回收价格；
- 数据来源主要是：
  - `quote.cngold.org`
  - `api.jijinhao.com`
- 按分类展示；
- 带更新时间与昨收价；
- 做了短时缓存。

此外，后端其实还提供了：

- `/api/widgets/metal-prices`：现货贵金属价格；
- `/api/widgets/chowtaifook-prices`：周大福价格；

但当前前端主要使用的是 `cngold-prices` 这一套数据。

## 3.4 我的页面：配置中心 + 数据管理中心

“我的”页面实际上承担了系统设置页的角色。

当前可做的事情很多：

### 3.4.1 孕期信息管理

- 设置末次月经日期；
- 设置宝宝昵称；
- 用于计算孕周、阶段与进度。

### 3.4.2 备份与恢复

- 可导出 JSON 备份；
- 可从 JSON 恢复；
- 备份中不仅包含胎动与孕期信息，还包含：
  - 体重；
  - 主题；
  - 动画速度；
  - AI 供应商与模型；
  - AI prompts；
  - 图片模式与压缩配置。

这说明备份不是“数据备份”而是 **应用状态快照**。

### 3.4.3 测试数据生成

- 可一键生成测试胎动数据和体重数据；
- 便于演示、调试趋势图与日历；
- 对开发体验很有帮助。

### 3.4.4 外观与交互偏好

- 深色 / 浅色主题；
- 首页气泡动画速度。

### 3.4.5 AI 配置中心

可配置内容相当全：

- AI 供应商：智谱 / 阿里云百炼；
- 各自模型名；
- system prompt；
- user prompt；
- 默认 prompt；
- 菜谱推荐 prompt；
- thinking 开关；
- 图片模式：URL / inline；
- 图片基准 URL；
- 图片压缩目标大小。

这意味着项目已经不是“硬编码接 AI”，而是具备一定的 **AI 可调参与实验能力**。

### 3.4.6 清空数据

- 会清空当前设备本地状态；
- 同时推送到服务端；
- 属于高风险操作，但对自用产品是必要功能。

## 4. 从源码视角看，项目由哪些部分组成

项目总体结构比较简单：

- `src/`：前端 React + TypeScript 单页应用；
- `server/`：Express 后端与 Python AI 辅助脚本；
- `bbcare.sqlite`：运行时 SQLite 数据库（默认在项目根目录）；
- `dist/`：前端构建产物（构建后生成）；
- `docker-compose.yml` / `Dockerfile`：容器化部署。

### 4.1 前端结构

前端目前属于 **“一个大页面文件 + 少量领域模块”** 的组织方式。

#### 关键文件

- `src/App.tsx`
  - 主应用入口；
  - 几乎包含全部页面组件与大部分页面逻辑；
  - 包括：首页、统计、小工具、我的、弹层、抽屉、状态同步逻辑。
- `src/api.ts`
  - 前端调用后端 API 的统一封装；
  - 定义了远端状态、菜单、AI、金价等请求。
- `src/storage.ts`
  - 本地 `localStorage` 状态加载、保存、清空；
  - 兼容老版本 key；
  - 负责默认值和规范化。
- `src/routes.ts`
  - 基于 hash 的轻路由。
- `src/useHashRoute.ts`
  - 路由监听 hook。
- `src/domain/`
  - 轻量领域模型：
    - `movement.ts`：胎动事件、按小时聚合、有效胎动规则；
    - `pregnancy.ts`：孕周与孕期阶段计算；
    - `weight.ts`：体重记录规范化、增删改查辅助。
- `src/styles.css`
  - 全局样式；
  - 移动端 UI 的大部分视觉表现都在这里。

#### 前端特点

- 没有引入 React Router，而是 hash 路由；
- 没有 Redux / Zustand 等状态管理；
- 状态大多集中在 `App.tsx` 顶层；
- 页面组件虽然分函数，但依然都在一个超大文件里；
- 优点是上手快、心智简单；
- 缺点是随着功能增加，维护成本会快速上升。

### 4.2 后端结构

后端也同样是“轻量但集中”。

#### 关键文件

- `server/index.js`
  - 后端主文件；
  - 包含数据库初始化、状态读写、日志、菜单、金价、上传、AI 接口等全部逻辑。
- `server/ai_caller.py`
  - Python 侧 AI 调用器；
  - 负责与智谱 / DashScope 兼容接口交互；
  - 主要用于图片问答调用场景。
- `server/read_menu_excel.py`
  - 用于把 Excel 菜单解析成 JSON；
  - 说明菜单数据来源最初是表格，而不是手写 JSON。
- `server/daily_menu_data.json`
  - 当前实际使用的菜单数据。

#### 后端特点

- 使用 `better-sqlite3`，同步调用、实现直接；
- 所有 API 写在一个文件里；
- 逻辑型后端，不依赖 ORM；
- 对个人项目而言实现成本低；
- 对继续扩展而言，模块边界开始显得不够清晰。

## 5. 核心数据模型

## 5.1 胎动事件

当前最核心的数据实体是 `MovementEvent`：

- `id`
- `type`
- `ts`

它是非常轻的事件模型，优点是：

- 追加式记录方便；
- 做小时聚合方便；
- 可支持未来更复杂统计。

### 5.1.1 胎动事件限制

当前会对事件数做上限控制：

- 前端 / 后端规范化时都会截断；
- 当前上限大约是 **2000 条事件**。

这代表一个很重要的事实：

- 现在项目更偏近期记录工具；
- 还不是“无限长期全量历史档案”。

## 5.2 体重记录

`WeightRecord` 结构更偏“按天聚合值”：

- `date`
- `weightKg`
- `updatedAt`

特点：

- 每天一条；
- 同一天后写覆盖前写；
- 会做日期校验和体重范围校验；
- 当前默认最多保留约 730 条（约两年量级）。

## 5.3 孕期信息

`PregnancyInfo` 核心上比较简单：

- `lmpDate`
- `babyName`

系统由此计算：

- 孕周；
- 剩余天数；
- 阶段（孕早 / 中 / 晚期）；
- 280 天进度百分比。

## 5.4 AppState / RemoteState

项目真正的“总状态”并不只是业务数据，还包括偏好与 AI 配置。

大致包含：

- `pregnancyInfo`
- `events`
- `weights`
- `updatedAt`
- `bubbleSpeed`
- `themeMode`
- `aiVendor`
- `aiModelZhipu`
- `aiModelAliyun`
- `aiSystemPrompt`
- `aiUserPrompt`
- `aiUserPromptDefault`
- `aiMenuRecipeSystemPrompt`
- `aiMenuRecipePrompt`
- `aiMenuRecipePromptDefault`
- `aiThinking`
- `aiImageBaseUrl`
- `aiImageMode`
- `aiImageTargetKb`

这说明：

- 这个项目已经不是“几个字段的小 Demo”；
- 它有一套比较完整的用户侧应用状态模型。

## 6. 数据存储与同步机制

这是项目里非常关键的一部分。

## 6.1 本地存储

前端会把状态放进 `localStorage`。

特点：

- 启动快；
- 离线也能看到上次状态；
- 对单用户个人场景很合适；
- 兼容旧版 key，并逐步迁移到统一的 `KEY_STATE`。

## 6.2 服务端存储

后端用 SQLite 存储两张表：

- `state`
  - 只有一行主状态（`id = 1`）；
  - 以 JSON 字符串保存整个应用状态；
- `logs`
  - 存客户端日志、AI 调用日志、上传日志等。

### 6.2.1 一个非常重要的事实

当前服务端状态不是“每个用户一份”，而是：

- **整个服务实例共享一份状态**；
- 所有客户端都在读写同一个 `state.id = 1`。

这代表项目目前本质上是：

- **单用户 / 单家庭 / 单实例** 设计；
- 如果直接公网开放给多人使用，会互相覆盖数据。

这是当前架构最重要的边界之一。

## 6.3 本地与远端同步

同步策略大致是：

- 启动时：
  - 拉服务端状态；
  - 比较 `updatedAt`；
  - 谁新用谁；
- 使用中：
  - 本地状态变化先写 `localStorage`；
  - 再延时推送到服务端；
- 首页可手动下拉刷新：
  - 再次比较远端状态与本地状态；
  - 决定拉取还是回推。

### 6.3.1 这个同步方案的优点

- 简单；
- 心智成本低；
- 对少量设备同步已经够用；
- 不需要复杂冲突解决。

### 6.3.2 这个同步方案的代价

它本质上是 **last-write-wins（最后写入覆盖）**：

- 两台设备同时改，后写可能覆盖先写；
- 没有字段级合并；
- 没有事件级合并；
- 没有用户维度隔离。

如果未来真要多设备稳定使用，这里会是第一批必须重构的地方。

## 7. 后端 API 全景

按功能可以这样理解：

### 7.1 基础状态与日志

- `GET /api/health`
  - 健康检查。
- `GET /api/state`
  - 获取全局应用状态。
- `PUT /api/state`
  - 覆盖写入全局应用状态。
- `POST /api/logs/client`
  - 写客户端日志。
- `GET /api/logs`
  - 查看日志列表。

### 7.2 菜单与 AI 菜谱建议

- `GET /api/widgets/daily-menu`
  - 返回指定日期映射出的每日菜单。
- `POST /api/widgets/menu-recipe`
  - 让 AI 根据餐次/食材推荐菜。

### 7.3 金价能力

- `GET /api/widgets/metal-prices`
  - 现货贵金属价格；
  - 优先用 Metals.Dev，未配置则回退到 Jijinhao。
- `GET /api/widgets/cngold-prices`
  - 实物黄金 / 金条 / 回收价；
  - 页面解析 + 实时报价补齐。
- `GET /api/widgets/chowtaifook-prices`
  - 周大福价格；
  - 当前后端有，前端暂未明显使用。

### 7.4 AI 相关

- `GET /api/widgets/ai-config`
  - 查看当前 AI vendor 是否已配置。
- `POST /api/uploads/image`
  - 上传图片到服务端 `uploads` 目录。
- `POST /api/widgets/food-check`
  - 图片食物识别与孕妇可食性问答。

## 8. AI 能力设计思路

## 8.1 支持多个供应商

当前设计不是写死单一模型，而是抽象成：

- 智谱（`zhipu`）
- 阿里云百炼 / DashScope（`aliyun`）

这让项目具备：

- 供应商切换能力；
- Prompt 调试空间；
- 成本 / 稳定性上的可替代性。

## 8.2 两类 AI 使用场景

### 8.2.1 图片问答：食物能不能吃

偏视觉理解 + 规则输出。

### 8.2.2 文本问答：菜单推荐菜

偏轻量文本生成。

这是一个合理拆分，因为它们对 prompt、超时、最大 token 的要求不同。

## 8.3 Python AI 调用器的意义

后端除了 JS 直接调模型，还提供 `server/ai_caller.py`。

这带来几个好处：

- 更容易在 Python 里兼容不同厂商接口细节；
- 便于图片请求、fallback、重试等处理；
- 以后如果要做更复杂的 AI 预处理，Python 脚本是一个自然扩展点。

## 8.4 图片提交的两种模式

### URL 模式

优点：

- 更接近很多大模型文档示例；
- 请求体更小。

风险：

- 模型服务必须能访问该 URL；
- 需要一个正确、可访问的 `aiImageBaseUrl`；
- 在内网 / Docker / 反向代理场景下更容易踩坑。

### Inline 模式

优点：

- 不依赖外部访问 URL；
- 本地或私有部署更稳。

缺点：

- 请求体更大；
- 对图片压缩更敏感。

项目当前同时支持两种模式，这一点设计得很实用。

## 9. 金价能力的实现思路

这部分虽然是“副功能”，但实现并不敷衍。

### 9.1 数据来源策略

- 如果配置了 `METALS_DEV_API_KEY`，现货贵金属优先使用 Metals.Dev；
- 否则从 `api.jijinhao.com` 拉现货报价；
- `cngold` 页面负责提供“有哪些金店/金条/回收分类和条目”；
- `jijinhao` 再补充这些条目的实时价格数据。

### 9.2 工程特点

- 做了 HTML 文本解析；
- 做了短时缓存；
- 做了数据清洗与字段统一；
- 前端呈现时按分类切表展示。

对一个个人项目来说，这部分已经算“有点像产品了”。

## 10. 菜单数据的来源与维护方式

当前菜单能力不是纯手写，而是经过“表格 → JSON”的转换。

### 10.1 当前数据链路

- 原始数据来源应是 Excel；
- `server/read_menu_excel.py` 负责解析工作簿中的 `Week1 ~ Week16`；
- 输出标准 JSON；
- 运行态由 `server/daily_menu_data.json` 直接读取；
- 后端带 `mtime` 缓存，文件不变则直接复用内存。

### 10.2 这套方式的优点

- 内容维护和程序逻辑分离；
- 非程序员也可以先改表格；
- 上线时只需要 JSON 文件，不依赖 Excel 解析环境。

### 10.3 当前限制

- 菜单是静态数据集；
- 没有用户画像；
- 没有禁忌项过滤；
- 没有和孕周、体重、血糖、过敏等条件联动。

## 12.2 工程层面的优点

## 13. 当前最重要的限制与风险

这部分非常值得正视。

## 13.1 当前是“单实例共享状态”，不是多用户系统

这是第一大限制。

如果将服务开放给多人：

- 所有人共享一份状态；
- 备份恢复会互相覆盖；
- 清空数据会影响所有人；
- 日志也没有用户隔离。

**结论：当前版本适合自用，不适合直接当公网多用户产品。**

## 13.2 缺少认证与权限控制

当前看不到账号体系，也没有鉴权：

- `/api/state` 可直接读写；
- `/api/logs` 可直接查看；
- `/api/uploads/image` 可直接上传；
- 上传目录是静态公开的。

如果部署到公网，这是明显的安全边界问题。

## 13.3 敏感配置管理风险

仓库中的 `.env` 当前包含真实样式的密钥内容，这是非常需要优先处理的信号。

无论这些 key 是否仍有效，都建议：

- 立即视为已泄露处理；
- 尽快轮换；
- 不再把真实密钥提交进仓库；
- 使用 `.env.example` 代替。

这是当前最值得优先行动的工程问题之一。

## 13.4 `App.tsx` 与 `server/index.js` 过大

这不是“现在不能跑”的问题，而是“继续加功能会越来越难”。

风险包括：

- 一个点改动容易影响别处；
- AI 难以局部理解；
- 很难给单个功能写测试；
- 代码 review 成本会越来越高。

## 13.5 同步冲突处理较弱

现在主要靠 `updatedAt` 决定谁覆盖谁。

这在以下场景会有问题：

- 两台手机同时记录；
- 离线使用后再上线；
- 一台设备恢复备份，另一台设备继续记录。

## 13.6 历史数据容量有限

胎动事件上限约 2000 条，意味着：

- 时间一长会丢较老数据；
- 长期统计和产前全程归档不够稳；
- 如果未来想做月/孕周级分析，容量可能很快不够。

## 13.7 菜单能力仍是“静态参考库”

每日菜单很有价值，但当前还不是“个性化营养系统”：

- 没和真实孕周严格绑定；
- 没有营养目标联动；
- 没有禁忌/过敏/妊糖条件；
- 没有购物清单 / 一键替换。

## 13.8 AI 输出仍属辅助，不可医疗化表述

图片问答和菜单推荐都很实用，但本质仍是大模型输出：

- 可能看错图片；
- 可能答得过度自信；
- 可能忽略个体禁忌症；
- 不应被误认为医疗建议。

如果以后继续产品化，文案和边界提示要更明确。

## 14. 我最建议优先做的改进

如果按性价比排序，我会这样建议。

## 14.1 第一优先级：安全与部署边界

### 建议 1：立刻处理密钥泄露风险

动作建议：

- 把 `.env` 从版本库敏感信息中清掉；
- 增加 `.env.example`；
- 轮换现有 API key；
- 补充 `.gitignore`。

### 建议 2：明确“仅单用户自用”或升级成“多用户系统”

现在不要停留在模糊状态。

二选一即可：

- **方案 A：明确定位为单用户自用工具**
  - 文档里明确说明；
  - 最好只部署在内网 / 私有环境；
  - 甚至可以给 API 加一个简单访问口令。
- **方案 B：开始向多用户演进**
  - 增加用户表；
  - 状态按用户隔离；
  - 上传和日志按用户归属；
  - `/api/state` 从“单行 JSON”改为“用户态存储”。

如果短期不准备做账号系统，我建议先选方案 A，避免误用。

## 14.2 第二优先级：拆模块

### 建议 3：把 `App.tsx` 拆成页面与组件

建议先按页面拆：

- `pages/HomePage.tsx`
- `pages/StatsPage.tsx`
- `pages/WidgetsPage.tsx`
- `pages/MinePage.tsx`
- `components/TopNav.tsx`
- `components/BottomNav.tsx`
- `components/DetailSheet.tsx`
- `components/DayDrawer.tsx`

不需要一次性大重构，先做“物理拆文件”就已经很有价值。

### 建议 4：把后端按路由模块拆分

例如：

- `server/routes/state.js`
- `server/routes/logs.js`
- `server/routes/widgets-menu.js`
- `server/routes/widgets-gold.js`
- `server/routes/widgets-ai.js`
- `server/lib/db.js`
- `server/lib/ai.js`
- `server/lib/menu.js`

这样以后改 AI 或金价，不会总在一个 1400+ 行文件里找。

## 14.3 第三优先级：提升数据可靠性

### 建议 5：把胎动从“整包 state JSON”里拆出来

如果继续长期用，胎动数据值得独立存表，比如：

- `movement_events`
- `weight_records`
- `app_settings`
- `logs`

好处：

- 更适合长期增长；
- 更容易筛选 / 聚合；
- 更容易做用户隔离；
- 更容易做冲突合并。

### 建议 6：同步改成“事件追加 + 设置覆盖”双通道

例如：

- 胎动事件：append-only；
- 设置项：按字段更新；
- 体重：按 `date` upsert；
- 最终再做服务端合并。

这会比当前整包覆盖稳很多。

## 14.4 第四优先级：让工具页更成体系

### 建议 7：菜单与孕周联动

当前最自然的升级方向是：

- 菜单按当前真实孕周优先匹配；
- 如果没有对应周，再 fallback 到循环菜单；
- 可增加“忌口 / 过敏 / 控糖 / 高蛋白”过滤。

### 建议 8：拍照问 AI 增加结果结构化

与其只返回长文本，不如要求模型返回结构化 JSON，再渲染成卡片：

- 食物名；
- 是否建议食用；
- 风险等级；
- 主要原因；
- 可食用建议量；
- 替代建议。

这样更稳定，也更方便以后做历史记录。

### 建议 9：给工具页增加“历史记录”

尤其适合：

- AI 食物识别历史；
- 菜单推荐历史；
- 金价关注项历史。

不然这些工具目前比较像“一次性查询”。

## 14.5 第五优先级：补文档与测试

### 建议 10：补一份真正反映现状的 README

当前 README 已经不能完全代表代码现状，至少这些内容应该补齐：

- AI 图片问答；
- 每日菜单；
- 菜谱推荐；
- 金价数据源；
- 单用户共享状态这一重要边界；
- 运行所需环境变量。

### 建议 11：先补最小单元测试

优先测纯函数：

- `effectiveCount`
- `groupByHour`
- `computePregnancy`
- `normalizeWeightRecords`
- 菜单日期映射逻辑

这些函数都很适合先补测试，成本低、收益高。

## 15. 如果让 AI 继续接手，这个项目最该先理解什么

如果以后再让 AI 来继续维护，我建议它先看这几个点：

1. **`src/App.tsx` 的顶层状态流**
   - 这是整个前端的真实控制中心。
2. **`src/storage.ts` 的本地状态结构**
   - 关系到兼容性、默认值和迁移。
3. **`server/index.js` 的 `readState/writeState`**
   - 这是前后端同步的核心。
4. **`src/domain/movement.ts`**
   - 胎动统计规则在这里，不要随便改。
5. **AI 图片模式（URL / inline）**
   - 这是最容易在部署环境里出问题的点。
6. **“单实例共享状态”这个基本事实**
   - 如果 AI 忽略这一点，很容易提出不贴合现状的方案。
